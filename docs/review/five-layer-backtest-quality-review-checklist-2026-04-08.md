# 五层交易系统回测质量审查总清单

**日期**：2026-04-08  
**适用项目**：`daily_stock_analysis`  
**用途**：作为五层交易系统回测方案设计、代码实现、评审验收前的统一质量闸门（checklist），用于确保回测结果具备足够可信度，不会误导后续的规则优化方向。

---

## 一、使用说明

本清单不是讨论“策略好不好”，而是先审查：

> **回测系统本身有没有把事情做对。**

建议在以下场景使用本清单：

1. 设计新的回测数据模型时
2. 设计新的回测引擎时
3. 实现回测 API / service / repo 时
4. 评审回测统计报表时
5. 基于回测结果准备修改规则时

### 审查结果建议分级
- **PASS**：可继续推进
- **WARN**：可继续，但必须标注风险
- **FAIL**：不得进入下一阶段

---

## 二、A 组：回测输入数据审查

### A1. 决策快照是否真实可追溯
- [ ] 回测对象是否来自真实历史候选快照，而不是后验重算结果？
- [ ] 每条回测记录是否能追溯到明确的 `trade_date / run_id / candidate_id / code`？
- [ ] 五层字段（`market_regime / theme_position / candidate_pool_level / setup_type / entry_maturity / trade_stage`）是否保存的是当时快照值？
- [ ] `trade_plan` / AI 审核结果是否保存的是当时版本，而非后续修订版？

### A2. Snapshot 与 Replay 是否明确区分
- [ ] 是否明确标记这是 `historical_snapshot_backtest` 还是 `rule_replay`？
- [ ] 是否避免把 replay 结果冒充真实历史表现？
- [ ] 是否避免在同一份主统计里混合 snapshot 与 replay 结果？

### A3. 行情数据是否完整且对齐
- [ ] forward bars 是否与 `trade_date` 精确对齐？
- [ ] 是否处理了停牌、缺失日线、日期错位、重复 bar？
- [ ] 指数/板块/市场宽度数据是否与个股数据使用同一交易日口径？

### A4. 未来函数风险是否被排除
- [ ] 是否保证所有用于真实快照回测的字段在决策当时真实可知？
- [ ] 是否没有使用后验主线题材标签替代当时实时题材认知？
- [ ] 是否没有用未来结构确认结果反推当天 `setup_type` / `entry_maturity`？

### A 组判定
- 若 A1 或 A4 有任一核心项不通过：**FAIL**
- 若 Snapshot / Replay 未区分：**FAIL**

---

## 三、B 组：回测时机与模式审查

### B1. 回测时机是否定义清楚
- [ ] 当前回测是否明确属于：真实快照回测 / 规则重演实验 / 参数校准回测？
- [ ] 当前回测的目标是否清楚：描述真实表现，还是比较规则优劣，还是调参？

### B2. 回测窗口是否与信号类型匹配
- [ ] 买入类信号是否使用了适合的 forward windows（如 1/3/5/10 日）？
- [ ] 卖出类信号是否重点评估了风险规避与错卖率？
- [ ] 观望类信号是否重点评估了“避免错误出手”和“等待更优时机”？

### B3. 不同回测模式是否被错误混用
- [ ] 是否把线上真实历史表现和实验性 replay 表现混成一个总结论？
- [ ] 是否将调参实验结果错误解读为真实生产表现？

### B 组判定
- 若回测模式不清晰：**FAIL**
- 若不同模式混算一个主结论：**FAIL**

---

## 四、C 组：信号类型评估机制审查

### C1. 信号类型是否已分类
- [ ] 当前回测对象是否明确标记为：买入类 / 卖出类 / 观望类？
- [ ] 是否避免将 `watch / focus / stand_aside` 当成买入信号来评分？
- [ ] 是否避免将卖出信号当成“做空收益信号”来评分？

### C2. 评估器是否与信号类型匹配
- [ ] 买入类是否使用入场质量评估器？
- [ ] 卖出类是否使用风险规避质量评估器？
- [ ] 观望类是否使用时机过滤评估器？

### C3. 评分逻辑是否没有混用
- [ ] 是否没有用同一套收益评分函数同时评估买入、卖出、观望？
- [ ] 是否没有因为“后来上涨了”就简单判定观望信号错误？

### C 组判定
- 若信号类型未区分：**FAIL**
- 若买入/卖出/观望共用同一主评分函数：**FAIL**

---

## 五、D 组：执行假设与成交模型审查

### D1. 执行模型是否显式记录
- [ ] 是否明确标记使用的是 `conservative / baseline / optimistic` 哪一档？
- [ ] 是否在回测结果中保存了 `execution_model`？

### D2. 入场假设是否现实
- [ ] 收盘后确认的信号，是否避免按当天收盘价成交？
- [ ] 是否统一定义了 `entry_timing` 和 `entry_price_basis`？
- [ ] 涨停票是否考虑了“买不到”的情形？

### D3. 出场假设是否现实
- [ ] 是否考虑跌停卖不掉？
- [ ] gap 跳空穿越止损位时，是否按更差价格处理？
- [ ] 止损/止盈触发价格是否符合既定模型，而非理想值？

### D4. 日线模糊区是否保守处理
- [ ] 同日既触发止盈又触发止损时，是否使用统一保守规则？
- [ ] 是否避免在模糊情况下默认选对系统更有利的结果？

### D5. 观望类是否处理得当
- [ ] 观望类主结果是否不强行假设真实成交？
- [ ] 是否允许有“强行入场对照组”，但与主结果清晰区分？

### D 组判定
- 若执行模型未标记：**WARN / FAIL**（主统计中应判 FAIL）
- 若存在理想化成交且未标识：**FAIL**

---

## 六、E 组：统计指标审查

### E1. 是否使用了多维主指标
- [ ] 是否同时展示样本数、胜率、平均收益、中位数收益？
- [ ] 是否同时展示风险指标（如回撤、MAE、盈亏比）？
- [ ] 是否避免只用单一指标下结论？

### E2. 是否考虑结果分布而非只看均值
- [ ] 是否看了中位数、分位数或极端样本占比？
- [ ] 是否评估结果是否由少数极端样本支撑？

### E3. 是否考虑稳定性
- [ ] 是否按时间分段（如月度）检查结果稳定性？
- [ ] 是否检查了不同 market regime 下的一致性？
- [ ] 是否避免只看局部行情阶段结果？

### E4. 是否按信号类型和模式分桶统计
- [ ] 是否按买入/卖出/观望分桶？
- [ ] 是否按 snapshot / replay / calibration 分桶？
- [ ] 是否按 execution model 分桶？

### E 组判定
- 若只用单一指标作为主结论依据：**FAIL**
- 若未分桶直接做总胜率/总收益结论：**WARN / FAIL**

---

## 七、F 组：样本量与稳定性审查

### F1. 样本量是否达标
- [ ] 当前结论对应的样本量是否达到最低观察门槛？
- [ ] 若是组间比较，两边样本量是否都达标？
- [ ] 若要生成优化建议，是否达到更高样本量门槛？

### F2. 是否存在小样本误导
- [ ] 当前好结果是否可能只是少量样本噪声？
- [ ] 是否存在“只出现了几次但收益极高”的假强模式？

### F3. 是否存在时间不稳定
- [ ] 是否只在单一月份/单一行情下表现好？
- [ ] 是否跨阶段、跨环境仍然大致成立？

### F 组判定
- 若样本不足却输出强建议：**FAIL**
- 若稳定性未验证但已准备改规则：**FAIL**

---

## 八、G 组：归因与结论解释审查

### G1. 归因是否清晰
- [ ] 当前改善/恶化到底来自哪一层（环境 / 题材 / 股票池 / 买点 / 阶段 / 交易计划）？
- [ ] 是否避免把执行模型差异误判为规则差异？
- [ ] 是否做过必要的分层对比或消融分析？

### G2. 结论是否可追溯
- [ ] 每个关键结论能否回到具体样本？
- [ ] 是否能抽样查看某个结论背后的原始候选和 forward bars？
- [ ] 是否能解释“为什么得到这个建议”？

### G3. 输出是否分级
- [ ] 当前结论是否明确属于：Observation / Hypothesis / Actionable Recommendation？
- [ ] 是否避免把 Observation 直接升级为规则改动建议？

### G 组判定
- 若归因不清却要改规则：**FAIL**
- 若结论不可追溯：**FAIL**

---

## 九、H 组：优化建议闸门审查

### H1. 建议是否满足基本条件
- [ ] 样本量达标
- [ ] 多个主指标方向一致
- [ ] 稳定性不过差
- [ ] 归因相对清晰
- [ ] 已区分真实快照结论与 replay 实验结果

### H2. 建议是否经过复核
- [ ] 是否已经通过 replay 或参数实验做过复核？
- [ ] 是否与旧规则在同口径、同数据上比较过？

### H3. 是否触碰红线
- [ ] 是否试图基于小样本自动改核心门控规则？
- [ ] 是否试图基于单一指标直接推动规则变更？
- [ ] 是否试图把 optimistic 模型结论作为主优化依据？

### H 组判定
- 任一红线触发：**FAIL**

---

## 十、最终总判定模板

建议每次评审后输出：

### 回测质量审查结论
- A 组（输入数据）：PASS / WARN / FAIL
- B 组（时机与模式）：PASS / WARN / FAIL
- C 组（信号评估机制）：PASS / WARN / FAIL
- D 组（执行假设）：PASS / WARN / FAIL
- E 组（统计指标）：PASS / WARN / FAIL
- F 组（样本量与稳定性）：PASS / WARN / FAIL
- G 组（归因与解释）：PASS / WARN / FAIL
- H 组（优化建议闸门）：PASS / WARN / FAIL

### 总结论
- [ ] 可进入开发实现
- [ ] 可进入内部实验，但不得输出主结论
- [ ] 可进入规则优化候选池
- [ ] 不得推进，需先修正回测质量问题

---

## 十一、一句话结论

> 五层交易系统的回测，不是“能跑出收益统计”就够了，而是必须先通过输入真实性、时间一致性、信号分类、执行现实性、统计稳健性和建议闸门这几道质量检查；只有过了这些闸门，回测结果才配拿来指导规则优化。
